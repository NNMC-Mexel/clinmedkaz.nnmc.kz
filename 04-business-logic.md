# 04 · Бизнес-логика (целевая, to-be)

Документ описывает, как платёжная логика **должна** работать. Где целевое расходится
с текущим кодом — стоит пометка ⚠️ и ссылка на [05-code-review.md](05-code-review.md).

## Принципы

1. **Invitation-only.** Заказ нельзя создать без валидного, не отменённого приглашения.
2. **Снимок цены.** Цена фиксируется в приглашении и не меняется при колебаниях курса.
3. **Идемпотентность платежа.** Повторные действия (двойной сабмит, повтор колбэка,
   ретрай Halyk) не создают дублей и не ломают финальный статус.
4. **`paid` — финал.** Никакой колбэк не может «откатить» оплаченный заказ. Поздние
   колбэки только дополняют данные/аудит.
5. **Разделение статусов.** Платёж (`order.status`), публикация (`publicationStatus`),
   возврат (`refundStatus`), бухгалтерия (`accountingStatus`) и приглашение
   (`invitation.status`) — независимые измерения. ⚠️ Сейчас платёж и приглашение смешаны.
6. **Сверка подлинности и суммы.** Колбэк применяется, только если совпал `secret_hash`
   **и** сумма/валюта соответствуют заказу. ⚠️ Сейчас сумма не сверяется.
7. **Аудит всего.** Каждый колбэк сохраняется, включая несопоставленные и отклонённые.

## Поток 1. Создание приглашения (админ)

**Вход:** email, ФИО, телефон, название статьи, язык, флаг «отправить письмо».

**Шаги:**
1. Валидация: корректный email, название статьи ≥ 3 символов.
2. Создать `invitation` со `status = created` и снимком цены (`invitationPricingSnapshot`).
3. Сформировать ссылку `BASE_URL/?invite=<id>&lang=<lang>`.
4. Если включена отправка — выслать письмо автору.

**Целевые улучшения:**
- ⚠️ Создание приглашения и отправка письма должны быть **раздельными** шагами: если SMTP
  упал, приглашение уже создано, и админу нужно явное «письмо не ушло, отправить повторно»,
  а не общая ошибка. Сейчас исключение в `sendMail` всплывает как ошибка всей операции.
- Действия над приглашением: **повторно отправить письмо**, **отменить** (`cancelled`),
  **скопировать ссылку**.
- Защита от дублей: предупреждать, если для той же статьи/почты уже есть активное приглашение.

## Поток 2. Создание заказа (автор отправил форму)

**Вход:** `invitationId`, ФИО, email, телефон, резидентство (название статьи берётся из
приглашения, не из формы).

**Правила (как должно быть):**
1. Приглашение существует и не `cancelled`.
2. Если приглашение уже оплачено (или есть оплаченный заказ) → ошибка «уже оплачено».
3. Если по приглашению есть **активный** заказ (`created`/`token_issued`) → вернуть его
   `payUrl`, **не создавать новый** (идемпотентность).
4. Иначе создать заказ: рассчитать цену по резидентству из снимка, сгенерировать
   `invoiceId` и `secretHash`, статус `created`.
5. Перевести приглашение в `payment_started`.

**Целевые улучшения:**
- ⚠️ Убрать `payment_started` из набора «активных» статусов заказа **или** реально
  присваивать его заказу — сейчас он только у приглашения (рассинхрон, F-2).
- Ограничить число повторных попыток/частоту (rate-limit), чтобы не плодить `invoiceId`.
- Серверная валидация телефона/почты с понятными сообщениями на языке формы.

## Поток 3. Инициация оплаты (выдача payment object)

**Шаги:**
1. `GET /api/payments/:id/payment-object`.
2. Проверки: заказ существует; не `paid`; не в терминально-неуспешном статусе.
3. Запросить OAuth-токен Halyk (`client_credentials`) с `invoiceId`, `secretHash`,
   `amount`, `currency`, `postLink`/`failurePostLink`.
4. Перевести заказ в `token_issued`.
5. Вернуть `paymentObject` для `window.halyk.pay()`.

**Целевые улучшения:**
- Логировать факт выдачи токена с маскированием секретов.
- Тайм-аут и ретраи на запрос токена Halyk; понятная ошибка пользователю при недоступности.

## Поток 4. Обработка результата (postlink)

`POST /api/halyk/postlink` — серверный колбэк Halyk (источник истины о результате).

**Алгоритм (целевой):**
1. Сохранить **сырой** колбэк в `callbacks` (всегда, даже если заказ не найден).
2. Найти заказ по `invoiceId`. Если нет — записать `matched=false` и выйти (200 OK).
3. Добавить запись в `order.postbacks` для аудита.
4. Если заказ уже `paid` → **не менять статус**, только дозаполнить
   `halykReference`/`cardMask` при отсутствии. (Идемпотентность, защита финала.)
5. Сверка подлинности:
   - `secret_hash` совпал с `order.secretHash`? Иначе → `postlink_rejected`.
   - ⚠️ **сумма и валюта** в колбэке совпали с заказом? Иначе → `postlink_rejected`
     (сейчас этой проверки нет — F-4).
6. По `code`: `ok` → `paid`, иначе → `failed`.
7. При `paid`: проставить `paymentReceivedAt`, `publicationStatus = ready_to_publish`,
   синхронизировать приглашение, отправить письмо админу.

**Целевые улучшения:**
- ⚠️ Аутентифицировать сам endpoint (IP-allowlist Halyk / общий секрет в URL), а не
  только сверять `secret_hash` в теле — F-3.
- Сравнение `secret_hash` через `timingSafeEqual`.
- Дедупликация колбэков по идентификатору транзакции.

## Поток 5. Пострасчётный workflow (бэкофис) — *целевое, пока не реализовано*

После `paid` заказ живёт в нескольких независимых измерениях. Нужны endpoint'ы и UI:

- **Публикация:** `ready_to_publish → published` (+ `articleUrl`, `doi`, `publishedAt`),
  либо `on_hold` / `cancelled`.
- **Возврат:** `none → requested → approved|rejected → processed` (+ `refundReason`).
- **Бухгалтерия:** `new → reconciled → exported → disputed`.
- **Комментарий админа** к заказу.

Каждое изменение пишет `updatedAt` и желательно — запись в журнал действий (кто/что/когда).

## Сводная таблица переходов заказа

| Из | Событие | В | Условие |
|----|---------|---|---------|
| — | форма отправлена | `created` | валидное приглашение, нет активного заказа |
| `created` | выдан токен Halyk | `token_issued` | заказ не оплачен/не закрыт |
| `token_issued` | postlink, secret+сумма OK, code=ok | `paid` | финал |
| `token_issued` | postlink, secret+сумма OK, code≠ok | `failed` | можно создать новый заказ |
| любой ≠ `paid` | postlink, secret/сумма НЕ OK | `postlink_rejected` | аудит |
| `paid` | повторный postlink | `paid` | только дозаполнение полей |
| `created`/`token_issued` | админ отменил | `cancelled` | *целевое* |
| `paid` | возврат проведён | `refunded` | *целевое* |

Полные диаграммы и поля — в [03-data-model.md](03-data-model.md).
