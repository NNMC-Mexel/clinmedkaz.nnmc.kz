# 09 · Отчёт бизнес-аналитика по платёжной логике

Дата: 2026-06-26. Объект: сервис приёма оплаты за публикацию статей ClinMedKaz.
Анализ проведён по фактическому коду после рефакторинга (см. [07-changelog.md](07-changelog.md)).

---

## 1. Executive summary

Сервис закрывает основную бизнес-задачу — **принять оплату за публикацию принятой статьи и
дать редакции контроль над платежами**. После технического рефакторинга ядро платежа
(создание заказа, токен Halyk, обработка колбэка) стало **корректным и безопасным**:
оплата идемпотентна, сверяется сумма, колбэк защищён, статусы заказа и приглашения разведены.

Однако с точки зрения **бизнес-процесса целиком** продукт ещё не «зрелый». Главные смысловые
пробелы лежат не в приёме денег, а вокруг него:

- **Возврат средств — фиктивный**: статус возврата есть, но платёж остаётся `paid`, а реальная
  операция возврата нигде не происходит (только ручной банк).
- **Плательщик не получает подтверждение/чек** — риск споров и chargeback.
- **Нет защиты от двойной оплаты на уровне статьи** (только на уровне ссылки).
- **Резидентство выбирает сам плательщик** — это определяет валюту и сумму без проверки.
- **Учёт в мульти-валюте** (KZT/USD) не консолидирован для отчётности.
- **Нет журнала действий администраторов** — при нескольких админах теряется подотчётность.

Оценка зрелости процесса: **«работает, но требует достройки до финансово-надёжного»**.
Ни один из пробелов не блокирует приём денег, но несколько из них создают финансовый и
репутационный риск. Приоритеты — в разделах 7–8.

## 2. Границы анализа

В скоупе: процесс «приглашение → оплата → результат → пострасчётный учёт/публикация»,
бизнес-правила, краевые сценарии, отчётность, соответствие. Вне скоупа: инфраструктура,
качество кода (см. [05-code-review.md](05-code-review.md)).

Акторы: **Автор/плательщик**, **Администратор/редакция**, **Бухгалтерия**, **Halyk ePay**.

## 3. Бизнес-процесс as-is (end-to-end)

```
[Редакция] принимает статью
     │
     ▼
1. Создание приглашения (персональная ссылка) ── снимок цены (USD-fee, курс, KZT/USD суммы)
     │  email автору (опционально)
     ▼
2. [Автор] открывает ссылку → заполняет ФИО/email/телефон, выбирает резидентство
     │  (название статьи берётся из приглашения, не редактируется)
     ▼
3. Создание заказа → расчёт суммы по резидентству из снимка
     │
     ▼
4. Оплата картой в Halyk ePay (3D Secure, данные карты на стороне банка)
     │
     ▼
5. Halyk postlink → сверка secret+сумма → заказ paid/failed → приглашение paid
     │  email админу об оплате
     ▼
6. [Редакция] в /admin: видит оплату, ведёт публикацию (DOI/URL), учёт, комментарии
```

Сильные стороны процесса: оплата только по приглашению, фиксация цены на момент приглашения,
PCI-нагрузка вынесена в банк, мультиязычность (ru/kk/en), юридические страницы присутствуют,
аудит колбэков ведётся.

## 5. Анализ краевых сценариев (критичные для денег/клиента)

| Сценарий | Текущее поведение | Оценка |
|----------|-------------------|--------|
| Двойной сабмит формы / двойной клик | Возвращается существующий активный заказ, дублей нет | ✅ |
| Halyk повторяет postlink | Идемпотентно, `paid` не меняется | ✅ |
| Колбэк с неверной суммой | `postlink_rejected`, `reason=amount_mismatch` | ✅ |
| Оплата не прошла, автор пробует снова | Создаётся новый заказ, можно оплатить | ✅ |
| Halyk не прислал сумму в колбэке | Сумма «не проверена», платёж проходит | ⚠️ зависит от Halyk |
| Нерезидент, а USD-терминал не включён | Оплата упадёт на стороне Halyk | ⚠️ конфиг-риск |
| Плательщик-нерезидент выбрал «резидент РК» | Платит в KZT по снимку курса | ⚠️ самодекларация |
| Админ отменил приглашение, когда автор уже на странице оплаты | Заказ остаётся активным, автор **может доплатить**; postlink перезапишет приглашение в `paid` | ⚠️ отмена не распространяется на «в полёте» (G-7) |
| Две ссылки на одну статью, обе оплачены | Двойная оплата одной публикации | ❌ нет защиты (G-3) |
| Возврат средств | Меняется только флаг, платёж остаётся `paid`, деньги возвращаются вручную в банке | ❌ разрыв (G-1) |
| Приглашение «висит» месяцами | Цена/курс из старого снимка применяются как есть | ⚠️ нет срока годности (G-4) |
| Курс USD/KZT изменился | Используется статичный `USD_TO_KZT_RATE` из `.env` | ⚠️ ручное обновление |

## 6. Выявленные пробелы (gaps)

### Финансы и учёт
- **G-1. Возврат оторван от статуса платежа.** `refundStatus` можно выставить, но `order.status`
  остаётся `paid`, статус `refunded` нигде не присваивается, реальной операции возврата в
  Halyk нет. Итог: отчётность покажет оплаченным то, что фактически возвращено.
- **G-2. Мульти-валютная отчётность не консолидирована.** Заказы в KZT и USD лежат в одной
  колонке `amount`; нет агрегатов по валютам, нет суммирования в одной валюте для бухгалтерии.
- **G-9. Нет автосверки с выпиской Halyk.** `accountingStatus` ведётся вручную; нет импорта
  статемента/реестра для сверки «что пробито в банке = что в системе».

### Клиентский опыт
- **G-5. Плательщик не получает чек/подтверждение на email.** Письмо уходит только админу.
  Это повышает риск споров/chargeback и снижает доверие.
- **G-6. Нет данных для бухгалтерских/налоговых нужд плательщика** (квитанция, реквизиты,
  назначение платежа) автоматически.

### Операции
- **G-3. Нет дедупликации на уровне статьи.** Защита от двойной оплаты работает только в рамках
  одной ссылки; две ссылки на одну статью → потенциально двойная оплата.
- **G-7. Отмена приглашения не распространяется на уже созданный заказ** (см. сценарий выше).
- **G-8. Нет журнала действий админов (audit trail).** Изменения статусов/комментариев не
  фиксируют, *кто* их сделал. При нескольких администраторах теряется подотчётность —
  критично для финансовых операций и возвратов.

### Данные и соответствие
- **G-10. Нет политики хранения/удаления ПДн.** ФИО/email/телефон хранятся бессрочно; нет
  ретеншена и механизма удаления по запросу (закон РК о персональных данных).
- **G-11. Самодекларация резидентства** влияет на валюту/сумму без подтверждения — потенциальный
  вопрос при налоговой/валютной проверке.

### Отчётность
- **G-12. Нет управленческих метрик.** Конверсия «ссылка → оплата», доля неуспехов, среднее
  время до оплаты, выручка по периодам/валютам — не считаются (есть только сырой CSV).

## 7. Реестр рисков

| ID | Риск | Влияние | Вероятность | Приоритет |
|----|------|---------|-------------|-----------|
| R-1 | Возврат не отражается в статусе → искажение выручки/сверки | Высокое (деньги) | Средняя | **P0** |

## 8. Рекомендации (по бизнес-приоритету)

> ✅ **Статус (2026-06-26):** Реализованы пункты 1–4 и 10. См. раздел «Реализовано» ниже.

**P0 — закрыть финансовые разрывы**
1. **Связать возврат со статусом платежа** (R-1, G-1): при `refundStatus=processed` переводить
   заказ в `refunded`; в идеале — инициировать возврат через Halyk API (refund), а не только флаг.
   Отчётность должна различать «оплачено» и «возвращено».

**P1 — контроль и доверие**
2. **Чек/подтверждение плательщику на email** после успешной оплаты (R-3, G-5): номер инвойса,
   сумма, статья, реквизиты получателя.
3. **Журнал действий админов** (R-4, G-8): кто и когда менял статусы/возвраты/комментарии; показать
   в карточке заказа. Особенно важно теперь, при входе через Strapi и нескольких админах.
4. **Защита от двойной оплаты статьи** (R-2, G-3): предупреждение/блок при создании второй
   активной ссылки на ту же статью (или связывание ссылок со статьёй как сущностью).

**P2 — учёт, соответствие, устойчивость**
5. **Консолидированная отчётность** (G-2, G-12): агрегаты по валютам/периодам, конверсия,
   доля неуспехов; экспорт, пригодный для бухгалтерии (раздельно KZT/USD).
6. **Срок годности приглашения** (R-6, G-4): авто-истечение и/или переоценка снимка цены.
7. **Автосверка с выпиской Halyk** (R-8, G-9): импорт реестра, авто-простановка `reconciled`.
8. **Политика хранения ПДн** (R-5, G-10): ретеншен, удаление по запросу.
9. **Уточнить обработку резидентства/валюты** (R-7, G-11): проверка поддержки USD-терминала;
   при необходимости — подтверждение резидентства или фиксация ответственности в оферте.
10. **Распространить отмену приглашения на активные заказы** (G-7): при отмене закрывать
    связанный незавершённый заказ.

## 9. Метрики для отслеживания (value-add BA)

- Конверсия: оплаченные / выданные приглашения.
- Доля неуспешных платежей и причины (`failed`/`postlink_rejected`).
- Среднее время «ссылка → оплата».
- Выручка по периодам и по валютам (KZT/USD раздельно).
- Доля и сумма возвратов.
- Время «оплата → публикация» (операционная эффективность редакции).

## 10. Реализовано (2026-06-26)

По итогам отчёта закрыты ключевые проблемы:

- **P0 · Возврат ↔ статус платежа (R-1/G-1).** При `refundStatus = processed` на оплаченном
  заказе платёж переходит в `refunded` и проставляется `refundedAt`. Отчётность теперь
  отличает удержанную выручку от возвращённой. (Реальный возврат денег через Halyk API —
  следующий шаг; сейчас движение денег ручное в банке, статус системно зафиксирован.)
  Реализация: `refundOutcome` в [domain.js](server/src/domain.js), PATCH-обработчик в server.js.
- **P1 · Чек плательщику (R-3/G-5).** После успешной оплаты плательщику уходит письмо-чек
  (номер платежа, статья, сумма, дата, получатель) на его языке. `sendPayerReceipt` в server.js.
- **P1 · Журнал действий админов (R-4/G-8).** Все админ-операции (создание/повтор/отмена
  приглашения, изменение заказа, возврат) и события оплаты пишутся в `auditLog` с актором
  (username из сессии Strapi/env или `api-token`). История видна в карточке заказа в `/admin`.
- **P1 · Защита от двойной оплаты статьи (R-2/G-3).** Оплата блокируется, если статья с тем
  же названием уже оплачена по другой ссылке; при создании приглашения админ видит
  предупреждение о дубле. `isArticleAlreadyPaid`/`findArticleDuplicates` в domain.js.
- **G-7 · Отмена распространяется на активный заказ.** Отмена приглашения закрывает связанный
  незавершённый заказ (`cancelled`).

Покрыто тестами (`npm test`, 20 тестов) и проверено end-to-end на JSON-хранилище.

**Осталось из рекомендаций:** реальный возврат через Halyk refund API (п.1, денежная часть),
консолидированная мульти-валютная отчётность и метрики (п.5), срок годности приглашений (п.6),
автосверка с выпиской Halyk (п.7), политика хранения ПДн (п.8), обработка резидентства/валюты (п.9).

### Вывод
Приём денег — надёжен. Чтобы процесс стал **финансово-зрелым**, в первую очередь нужно
закрыть «возврат ↔ статус платежа» (P0) и обеспечить **подотчётность + чек плательщику**
(P1). Остальное — про прозрачность учёта и соответствие. Реализация любого пункта не требует
переписывания ядра — оно к этому уже готово.
